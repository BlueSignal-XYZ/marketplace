# v1.1 Security Hardening Audit

**Date**: 2026-04-12
**Baseline**: 852 commits, pre-v1.1 merge
**Scope**: Firebase security rules, Cloud Function input validation, secrets
exposure, sensor ingestion, rate limiting, CORS.

## Methodology

1. Grep for all `process.env` / `import.meta.env` references to map the
   secrets surface.
2. Read `database.rules.json` and `firestore.rules` end-to-end; flag any
   wildcard writes or missing auth predicates.
3. Walk every `functions/**/*.js` HTTP handler for input-validation gaps.
4. Confirm `readings.js` validates device ownership AND sensor ranges
   before writing to `/readings/`.
5. Check CORS allow-list on the Cloud Functions Express app.
6. Confirm no committed secrets (API keys, service-account JSON, tokens).

## Findings

### ✅ SR-01: No secrets committed to the repo
Grep for `BEGIN PRIVATE KEY`, `AIza`, `sk_live`, `hubspot-prod-` across
history returned only example placeholders in `.env.example`. **No action.**

### ✅ SR-02: Firebase RTDB root rules default-deny
`database.rules.json:2-4` sets `.read: false, .write: false` at root, with
explicit per-node predicates below. Writes to undeclared paths fail. **No action.**

### ⚠ SR-03: Public-facing pre-order endpoint lacks rate limiting
`functions/preorder.js` accepts anonymous POSTs (no auth required by design).
Rate limiting relies on Firebase Cloud Functions default concurrency caps,
which are not per-IP. **Recommendation** (v1.2): add a lightweight per-IP
counter in RTDB with 60-second windows; reject after 10 req/min.

### ✅ SR-04: Sensor ingestion validates ownership + ranges
`functions/readings.js:28-44` checks API-key → device binding before write.
Sensor ranges enforced via `SENSOR_RANGES` at `functions/readings.js:10-23`
(pH 0–14, turbidity 0–4000 NTU, etc.). Readings outside range are tagged
`quality: 'out-of-range'` rather than silently accepted. **No action.**

### ⚠ SR-05: v2 device list formerly had no siteId/installerId filter
`functions/v2/devices.js listDevices` returned all user-owned devices;
SiteDetailPage and InstallerDashboard filtered client-side, wasting bandwidth
and exposing more rows than needed. **Fixed in v1.1**: server-side `siteId`
and `installerId` filters added with input validation (alphanumeric, ≤128
chars). Clients now pass filters via `getDevices(uid, { siteId })`.

### ⚠ SR-06: Credit generation had no audit trail
Prior: credits were written to `/credits/` without an immutable record of
inputs/formula. **Fixed in v1.1**: `/creditAuditLog/{creditId}` append-only
node captures readings, formula, tier, multiplier, result, and confidence
flags. Read-scoped to owner + admin. Write-scoped to server-side admin
with `!data.exists()` to prevent updates.

### ⚠ SR-07: Wallet-ownership signature verification missing
`functions/v2/blockchain.js:74` has a TODO to verify signature proves wallet
ownership. Without it, a user could claim any wallet address. **Priority P1**
for v1.1-B. Tracked in `CLAUDE.md` open items.

### ✅ SR-08: CORS allow-list explicit
Cloud Functions `index.js` sets CORS to the four known origins. No `*`
wildcard. **No action.**

### ⚠ SR-09: QR secret falls through to functions.config() default
`functions/qrcode.js:19` — `process.env.QR_SECRET || functions.config().qr?.secret`.
Not a bug, but document the source of truth in `secrets-inventory.md`.
**Action**: doc-only, no code change.

### ⚠ SR-10: Legacy configurator (`?app=landing`) analytics not instrumented
Plan (v1.1-B) to track traffic. If zero after 30 days, remove the tree in
v1.2. Flagged here because a deprecated code path accumulating complexity
is a supply-chain risk.

### ⚠ SR-11: Reading-ingestion accepts arbitrary `sensors` keys
`validateSensors` processes any key in the request body; unknown types
produce no `SENSOR_RANGES` entry, so no range check runs. Data gets stored
untagged. **Recommendation** (v1.2): reject unknown sensor keys outright,
or tag them `quality: 'unknown-type'` and exclude from credit generation.

### ✅ SR-12: Firebase Auth flow uses server-side token verification
v2 API client at `src/services/v2/client.ts:60-69` passes Firebase ID token
via `Authorization: Bearer`. Cloud Functions verify via the Firebase Admin
SDK. No JWT parsing on the client for authorization decisions. **No action.**
